# Elysia 项目总规划（可直接执行）

## 一、项目摘要
1. 项目目标：打造一个“低门槛记录 + 高价值探索”的情绪与思维网站，核心是让用户愿意开始写、持续写、并从记录中发现自我与共鸣。
2. 核心模块：`Elysia Home`（个人记录）与 `Elysia Universe`（公开探索），并通过 `MindMap` 连接情绪、金句、灵感、评论派生记录。
3. 体验原则：默认极简、渐进展开、一步可发、深度能力按需显现，避免传统记录软件“流程重、门槛高、懒得写”。
4. 技术基线：`Node.js + Fastify + PostgreSQL + Redis + Cloudflare R2 + WebSocket + OpenAPI 3.1`。
5. 部署基线：阿里云单机 `Docker Compose + Caddy + cloudflared(CF Tunnel)`；本地无 OAuth（开发假用户），线上接 LinuxDoConnect OIDC。
6. 视觉基调：水晶花液态玻璃风格，借鉴爱莉希雅“无瑕、纯净、温柔”的气质表达，不使用角色立绘素材。

## 二、产品与交互详细设计

### 2.1 Home（首要入口，反拖延核心）
1. 入口默认只有一个主动作：`写下此刻心情/思路`。
2. 首访使用 3 屏引导，可全程跳过并记住；回访仅保留一句温柔欢迎语。
3. 默认输入为“一句话文本”，只需一句即可发布。
4. 输入首句后字段智能逐步出现（每次 1-2 项）：金句、附加情绪、描述、图片/画布、时间、地点、公开设置。
5. 卡住引导策略：编辑空白停留 8 秒触发 1 条轻提示，同时常驻“给我灵感”手动按钮。
6. 自动保存：输入后 1.5 秒防抖保存草稿。
7. 草稿策略：每用户最多保留 5 份草稿，顶部小提示支持一键恢复上次。
8. 离线能力：支持离线写草稿，恢复网络后自动重试同步。
9. 发布反馈：短反馈动效后立即回到主视图，不强制填写全部字段。
10. 编辑规则：正文发布后 30 天内可改；公开/私密切换无时间限制。

### 2.2 Universe（星海探索，非列表沉浸）
1. 形态为无限二维上下左右探索，背景升级为深空蓝紫色流体星云（CSS 渐变与呼吸光晕）。
2. 每个视口固定 `1 主焦点 + 2 副焦点`。远处的非焦点内容化作纯白或粉色的悬浮水晶气泡。
3. 焦点更新节奏：视口停留 20 秒后微调，避免频繁跳动。
4. 显示策略：当用户“漫游”靠近时，远处的模糊气泡如深海上浮般逐渐清晰放大，展现出液态玻璃（Liquid Glass）质感。
5. 内容节奏：每日主题星团全天稳定，由半透明、折射光芒的“晶体飞鳐”（Canvas 绘制）在屏幕深处缓缓游动标示，飞鳐尾迹拖拽出细碎的粉色几何粒子；共鸣脉冲每 2 小时刷新。
6. 排序策略：热度 + 新鲜度混排，并保留中等个性化权重（40%）。
7. 互动方式：拖拽 hug/heart 到目标内容时，触发“粉色水晶飞花”的 Canvas 粒子爆发，并伴随水波纹荡漾。
8. 内容卡片默认精简展示：心情短语 + 金句 + 时间 + 分享者；详情再展开评论与派生关系。
9. 评论他人内容后自动生成“我的新记录”，并与原帖双向关联。

### 2.3 MindMap（记忆织网：往世乐土的记忆结晶）
1. 进入规则：首次进入网站默认 Home；若 30 分钟内有新记录，下次优先进 MindMap。
2. 用户控制：允许一键回 Home，并记忆入口偏好。
3. 模式切换：顶部一键 `简洁模式/深度模式`，并记忆上次模式。
4. 默认视图范围：本人记录 + 公开弱连接。
5. 布局算法：关系力导向 + 时间环叠加。
6. 节点设计（刻印化）：不再是普通气泡，而是悬浮的不规则水晶体：情绪节点为粉色多瓣水晶花；金句为浅金菱形星芒；普通记录为冰蓝/紫罗兰多面体。
7. 关系连线：使用带有微弱光子流动的金色记忆丝线连接各节点。
8. 清晰策略：高密度自动聚类时，表现为一颗巨大的“原石水晶”；点击或聚焦展开时，伴随视觉上的“震颤与绽放”动效，解聚成满屏的刻印节点。
9. 编辑策略：正文 30 天内可改；标签和关联永久可改；所有变更留审计轨迹。
10. 视图能力：关系图 + 时间轴 + 主题树可自由切换（简洁/深度）。

### 2.4 7天激活剧本（每天≤1分钟）
1. D1：一句话发布，建立“我可以马上记录”。
2. D2：补一项（仅金句或附加情绪二选一）。
3. D3：点选 1 条关联，形成第一条思维链。
4. D4：添加 1 张图或 1 段手写。
5. D5：浏览主题星团并完成一次互动。
6. D6：查看周轨迹并确认一个“本周关键词”。
7. D7：里程碑反馈（徽章 + 主题微动效 + 深度入口提示，不锁核心功能）。

## 三、UI 设计系统规范（给 Gemini 的实现基线）
1. 风格关键词：Crystal Bloom、Liquid Glass、Soft Glow、Calm Motion、Star Sea (星海)、Elysian Signets (乐土刻印)。
2. 颜色体系：冰晶白、晨雾蓝、珊瑚粉、浅金高光、深邃紫/星空蓝（Universe底色）；避免高饱和与重紫主导（除星海背景外）。
3. 材质体系：高斯模糊玻璃层、柔边高光、微色散描边、低噪点纹理背景。
4. 组件体系：记忆水晶节点、金色流光丝线、晶体飞鳐（Canvas）、液态卡片、渐进字段组、拖拽飞花表情。
5. 动效体系：主形变 260-320ms，次反馈 120-180ms，背景呼吸 6-10s 周期。纯视觉音符（♪）与水波纹涟漪反馈。
6. 信息层级：速记层永远在第一层，深度工具默认折叠在第二层之后。
7. 文案体系：爱莉希雅风格——温柔、热情、充满无瑕的爱。大量使用夸赞与拟声语气词（如“呀”、“呢”），句末恰当使用音符“♪”。
8. 三端适配：手机优先单手可达；iPad 强化手写分栏；PC 强化探索与多视图切换。
9. 可访问性：高对比模式、减弱闪烁开关、触控目标最小尺寸、关键操作可键盘触发。
10. 性能感知：所有大图先占位后清晰，禁止“白屏等待”。

## 四、系统架构与部署方案

### 4.1 服务拆分
1. API 服务：Fastify，负责鉴权、读写、权限、业务编排。
2. Worker 服务：处理图片转码、向量计算、周报汇总、导出任务。
3. Queue：异步任务编排与削峰。
4. DB：PostgreSQL 主存。
5. Cache：Redis（会话、热点、计数、限流、队列辅助）。
6. Object Storage：Cloudflare R2（图片、画布资源、导出文件）。
7. 反向代理：Caddy。
8. 外网入口：Cloudflare Tunnel（cloudflared）+ CF DNS。

### 4.2 环境策略
1. 本地：假用户切换，不启 OIDC，便于前后端联调。
2. 线上：LinuxDoConnect OIDC，完整会话与权限校验。
3. 发布策略：蓝绿或滚动重启；先灰度后全量。
4. 数据策略：每日全量备份 + 每小时增量（保留 3 天）+ 异常快照。

## 五、数据模型与存储策略
1. 用户域：`users`、`identities`、`sessions`、`user_preferences`。
2. 记录域：`records`、`record_quotes`、`record_emotions`、`record_links`、`drafts`。
3. 社交域：`comments`、`reactions`、`follows`（可选）、`notifications`。
4. 图谱域：`mindmap_nodes`、`mindmap_edges`、`insight_snapshots`。
5. 媒体域：`media_assets`、`drawing_docs`、`media_variants`。
6. 地点时间域：`locations`、`record_time_meta`。
7. 治理域：`reports`、`moderation_actions`、`sanction_logs`。
8. 统计域：`access_events`、`endpoint_minute_stats`、`user_frequency_stats`、`alert_events`。
9. 约束：单条最多 4 图，每图 ≤ 5MB，每用户全站最多 60 图。
10. 隐私：地点默认城市级打码，可手动更细；打码文本自然融入，不做生硬标签化。

## 六、API 与实时事件（对前端契约）
1. Auth：`/auth/oidc/login`、`/auth/oidc/callback`、`/auth/logout`、`/auth/dev/switch-user`。
2. Record：`POST /records`、`PATCH /records/:id`、`PATCH /records/:id/visibility`、`GET /records/:id`、`GET /home/feed`。
3. Comment/Derive：`POST /records/:id/comments`（返回派生记录 ID 与关联信息）。
4. Universe：`GET /universe/viewport`、`GET /universe/focus`、`GET /universe/hot`、`GET /universe/recent`。
5. MindMap：`GET /mindmap/me`、`GET /mindmap/:recordId`、`POST /mindmap/manual-link`。
6. Media：`POST /media/upload-sign`、`POST /media/complete`、`POST /drawings`、`GET /media/:id/variants`。
7. Insight：`GET /insights/emotion-trajectory`、`GET /insights/theme-evolution`、`GET /insights/resonance-network`。
8. Nudge：`GET /nudges/recommendations`、`PATCH /nudges/settings`、`POST /nudges/feedback`。
9. Governance：`POST /reports`、`GET /admin/reports`、`POST /admin/reports/:id/resolve`、`POST /admin/users/:id/sanction`。
10. Analytics：`GET /admin/analytics/overview`、`/endpoints`、`/users-frequency`、`/alerts`。
11. WebSocket 事件：`record.created`、`record.updated`、`reaction.updated`、`focus.shifted`、`moderation.changed`。

## 七、AI 与推荐能力边界
1. 明确禁止：AI 自动生成用户“描述/金句”正文。
2. 允许能力：标签建议、周报总结、轨迹洞察、主题聚类建议。
3. 接口模式：用户自填模型地址与密钥由客户端直连，不经服务器代发。
4. 举报机制：支持“疑似 AI 生成”举报，管理员确认后记次。
5. 处罚策略：3 次确认 7 天禁言，5 次封禁，审核 SLA 为 72 小时。
6. 关联算法：关键词规则 + 向量语义混合建链，实时增量 + 每日校准。

## 八、性能、负载与监控优化
1. 目标：Universe 首屏 ≤ 2.5s，核心交互 < 120ms。
2. 并发目标：500 DAU / 峰值 100 并发下稳定运行。
3. 异步化：图像处理、向量计算、导出、周报全走队列 Worker。
4. 缓存：热点内容与焦点节点 Redis 缓存，短 TTL + 主动失效。
5. 查询优化：关键查询加联合索引与分页游标，避免大偏移扫描。
6. 媒体优化：多尺寸 WebP/AVIF，先模糊占位再清晰。
7. 监控维度：全站、接口、用户访问频率三层。
8. 告警策略：阈值告警 + 突增检测。
9. 日志保留：明细 7 天，聚合 6 个月。
10. 看板：系统健康、Universe 热点、用户频率、风控异常、队列积压。
11. 限流：按用户 + IP 双维令牌桶。
12. 人机挑战：仅高频异常时触发 Cloudflare Challenge。

## 九、安全、隐私与治理
1. 认证授权：OIDC + RBAC（普通用户/管理员）。
2. 会话安全：短会话令牌 + 刷新机制 + 敏感操作二次校验（管理端）。
3. 上传安全：签名上传、类型校验、大小限制、恶意文件拦截。
4. 数据安全：敏感字段加密存储，最小权限访问。
5. 审计合规：管理员操作全审计，处罚有依据可回溯。
6. 隐私控制：公开开关默认记忆上次选择，用户可随时切换。
7. 个性化数据：默认开启并可关闭（站内行为 + 阅读停留偏好）。

## 十、测试计划与验收标准
1. 单元测试：长度规则、编辑期限、公开开关、图片配额、草稿限制、派生逻辑。
2. 集成测试：发布-补全-关联-评论派生-互动-举报全链路。
3. 契约测试：OpenAPI 与前端字段一致性、错误码与分页规则。
4. E2E 测试：手机/iPad/PC 三端关键路径。
5. 性能测试：视口加载、焦点刷新、WS 推送、图片切换与拼图详情。
6. 稳定性测试：断网续写、重连同步、队列拥塞恢复、缓存击穿保护。
7. 安全测试：OIDC 流程、权限绕过、限流、生效与告警。
8. 灾备演练：全量/增量恢复、异常快照恢复、Tunnel 中断恢复。

## 十一、里程碑与交付节奏
1. M1（后端基础）：鉴权、记录域、媒体上传、草稿与渐进流程、OpenAPI 初版。
2. M2（核心体验）：Universe 视口与焦点、MindMap 基础、评论派生、实时互动。
3. M3（深度能力）：洞察、7天激活、引导系统、导出、治理后台。

5. 每两周一轮 A/B：单轮控制 1-2 个变量，持续优化“愿意写”指标。

## 十三、阻塞问题优先修复（localhost 拒绝连接）
1. 启动前置检查：必须存在 `.env`，且 `DATABASE_URL`、`REDIS_URL`、`PORT`、`HOST` 配置有效（本地默认 `HOST=0.0.0.0`、`PORT=3000`）。
2. 启动顺序：`docker compose up -d postgres redis` -> `pnpm migrate` -> `pnpm dev`（后端）-> `pnpm --filter frontend dev`（前端）。
3. 访问地址：后端 `http://localhost:3000/api/healthz`，前端 `http://localhost:5173`，文档 `http://localhost:3000/docs`。
4. 失败即定位：若后端进程未启动，先检查 `.env` 缺失、端口占用、Node/Pnpm 环境；若前端空白，检查是否启动在其他端口。
5. 验收标准：浏览器可打开 `5173`，并能从前端触发发布请求，后端日志能看到 `/api/records` 请求记录。

## 十四、前端重构补充方案（Home + MindMap 强化）
1. 重构目标：在保持 `Crystal Bloom + Liquid Glass` 的基础上，显著强化“温柔、贴心、温暖、爱”的情绪表达，不做硬核工具化界面。
2. Home 背景规范：用户提供的是“说明链接”（Google imgres/TapTap 页面），实现时使用其对应舰桥背景图素材（可直接搬用原图）并叠加多层粒子/光晕/呼吸位移动效，形成“微动态”背景，禁止静态单图直铺。
3. Home 结构：保留一句话极速发布为首动作；补齐渐进字段（quote/extraEmotions/description/media/time/location/public）与 1.5s 防抖草稿。
4. Home 可发布闭环：接入 `POST /api/records`、`PATCH /api/records/:id/visibility`、`GET /api/home/feed`，发布成功后即时刷新 Home 流与 Universe 数据。
5. Universe 接口化：移除 Mock 卡片，改为读取 `GET /api/universe/viewport|focus|hot|recent`，保留“1主焦点+2副焦点”与拖拽反馈。
6. MindMap 重做：从占位页升级为可交互图谱视图，接入 `GET /api/mindmap/me`、`GET /api/mindmap/:recordId`、`POST /api/mindmap/manual-link`，支持简洁/深度模式切换。
7. 前端工程基线：建立 `api client + react-query + auth bootstrap + ws event bridge`，替代当前纯组件演示状态。
8. 主题与资产合规：开发与联调阶段允许直接使用用户指定原图；正式公开发布前补充来源与分发授权核对，若授权不可得再切换替代资产。
9. 视觉架构实现：采用 CSS3 + 纯 Canvas API 混合渲染架构。CSS3 负责高斯模糊、液态卡片、流体背景渐变以确保首屏性能；原生 Canvas API 专门封装为独立的 React Hook 或组件，负责飞鳐游动、流光连线、拖拽飞花粒子等高性能动效。如遇低端机型掉帧，平滑减少 Canvas 粒子数量，保留 CSS 玻璃质感。

## 十五、公开内容审核状态机（先审后发）
1. 总原则：用户选择“公开”后不应立即进入 Universe；内容需先过系统审核，图片优先人工审核。
2. 私密内容策略（放松审查）：选择不公开的文本内容采用放松策略，不走公开审核闸门；仅保留高危底线检测与风控留痕。
3. 图片策略（统一人工审查）：涉及图片内容均进入人工审查，但不公开内容不阻塞用户本人查看与整理。
4. 私密可见性：不公开内容可立即在“本人 Home / 本人 MindMap”可见；不得进入 Universe 与他人可见范围。
5. 文本状态机：`draft/private` -> `pending_auto_review` -> `pending_manual_review`（高风险/不确定）-> `approved_public` 或 `rejected_private`。
6. 图片状态机：带图片记录默认进入 `pending_manual_review`；管理员通过后才能变为 `approved_public`。
7. 编辑复审：已公开内容被编辑后状态变为 `pending_re_review`，复审通过前继续展示旧公开版本或临时下线（按策略开关）。
8. 风险命中规则：真实地址、联系方式、身份证件、精确定位、涉政/极端/反动词库与模型判定命中时转人工。
9. 审核结果：支持 `approve`、`reject`、`needs_changes` 三种结论；`needs_changes` 返回可读修改建议并允许用户重提审。

## 十六、管理员权限增强（治理后台升级）
1. 新增审核工作台：待审列表、风险标签、原文/媒体预览、历史版本、操作审计一体化。
2. 新增权限分级：`admin` 拆分为 `reviewer`、`senior_reviewer`、`super_admin`，限制高危操作。
3. 审核 API 新增：`GET /api/admin/moderation/queue`、`POST /api/admin/moderation/:id/approve`、`POST /api/admin/moderation/:id/reject`、`POST /api/admin/moderation/:id/request-changes`。
4. 复审与回滚：支持已公开内容紧急下线、复审重开、版本回滚、审计追踪。
5. 图片人工审核：支持批量判定、违规区域标注、证据留存。

## 十七、新人温馨引导（7天剧本落地补全）
1. 入口引导：首次登录展示 3 屏暖启动（可跳过、可回看），避免打断主流程。
2. 每日任务卡：Home 顶部展示“今日 1 分钟任务”，关联 `/api/onboarding/progress` 与 `/api/onboarding/complete-day`。
3. 引导触发器：空白停留 8 秒弹轻提示；手动“给我灵感”按钮固定可见；支持 `/api/nudges/recommendations`。
4. 里程碑反馈：D7 完成后展示徽章、微动效、MindMap 深度入口与回顾卡。
5. 失败保护：连续 2 天未完成时给“温柔重启”文案，不使用惩罚式提示。

## 十八、安全风险监控与保护强化
1. 新增监控维度：审核吞吐、待审积压、高风险命中率、误杀率、复审通过率、管理员处理 SLA。
2. 自动告警：待审超过阈值、敏感词突增、同 IP 高频提交、异常公开率时触发告警事件。
3. 保护策略：高风险用户进入强校验（Turnstile + 限流 + 审核优先级下调）；必要时临时冻结公开权限。
4. 数据审计：所有审核动作和规则命中记录写入 `audit_logs` 与 `moderation_actions` 扩展表，支持可追溯导出。
5. 测试门禁：新增“审核状态机、编辑复审、文本高风险拦截、图片人工审核、管理员误操作回滚”测试集。

## 十九、交付节奏重排（基于当前新增需求）

## 二十、AI 辅助审核（管理员可配置 GPT）
1. 触发方式：仅管理员后台可触发“扫描最近 1 小时公开申请（仅文字）”；可选支持定时任务。
2. 数据收集范围：默认仅收集 `visibility_intent='public'` 且 `publication_status in ('pending_auto','pending_manual')` 的“仅文字公开申请”。
3. 聚合格式：将 `user + record` 组装为 CSV（`record_id,user_id,username,mood_phrase,quote,description,tags,created_at`）。
4. 调用模式：管理员在后台设置 `baseUrl + apiKey + endpointType(responses|completions) + model`，后端代调用，不在前端直连。
5. 输出等级：`极低/低/中/较高/高/极高`，并返回风险标签与原因；后端保存原始响应与结构化结果。
6. 结果落地：
   - `极低/低`：若未命中敏感词/高危规则，则直接公开。
   - `中/较高`：进入人工审核队列（较高优先级更高）。
   - `高`：进入“管理员二次审查”队列，不自动公开。
   - `极高`：直接进入风控队列并触发 24h 风控限制。
7. 私密内容处理：私密文本默认不进入公开审核批次；如命中高危底线规则，仍可触发风控告警与人工复核。
8. 防滥用：`baseUrl` 强制 `https`，禁止内网地址；单次批处理按分片执行，避免超长上下文与接口超时。

## 二十一、涉政/血腥暴力高危处理与申诉闭环
1. 命中条件：规则命中或 AI 判定为 `高/极高` 且标签为涉政、血腥暴力、极端煽动。
2. 自动动作：立即拒绝公开、记录本次编辑快照；`极高` 优先进入 24h 风控队列（可由管理员裁决后升级为警告/封禁）。
3. 永久封禁条件（严格）：若判定为涉政且人工复核确认成立，或判定为血腥暴力严重违规，则对用户与 IP 执行永久封禁。
4. 证据留存：保存触发文本、命中规则、模型等级、审核链路、操作者和时间戳，支持审计导出。
5. 申诉入口：被封禁用户跳转申诉页，仅可提交申诉与查看状态。
6. 防重复申诉：存在 `pending` 申诉时禁止重复提交，后端返回“已提交请勿重复”，前端按钮置灰不可用。
7. 单次申诉规则：每个封禁事件仅允许一次申诉；若申诉被拒绝，该事件不再开放二次申诉。
8. 管理员处理：后台可查看申诉、证据与历史操作，支持驳回/解封（用户与 IP 可独立解封）。

## 二十二、严格封禁与限制发帖策略矩阵
1. `涉政 + 人工确认成立`：用户永久封禁 + IP 永久封禁；允许一次申诉。
2. `血腥暴力严重违规`：用户永久封禁 + IP 永久封禁；允许一次申诉。
3. `高危倾向（未达永久封禁）`：发出警告并进入限制发帖频率模式（限频 + 强校验 + 人工复核优先）。
4. `中高风险重复触发`：按阶梯加严（24h 冷却 -> 7 天限制公开 -> 临时封禁），达到阈值后升级永久封禁候选。
5. 频率限制建议：高危用户发布频率收紧为每小时 1 条、每天 3 条，评论互动同样降速并要求 Turnstile。
6. 解除条件：非永久封禁的限制到期后自动恢复；永久封禁仅在申诉通过后由管理员手动解封。
7. 风控队列规则：`极高` 直接进入 `risk_control_24h`，期间限制公开申请与高频互动；管理员可在后台执行 `解除 / 警告 / 封禁`。

## 二十三、管理员后台设计（仅 PC 端）
1. 适配范围：仅桌面端（建议最小宽度 1280），不做移动端。
2. 模块划分：
   - 审核总览（待审量、风险分布、SLA、封禁趋势）
   - 准入申请池（首次加入申请审核、建议模板、批处理）
   - 审核队列（文本/图片、优先级、认领、批处理）
   - AI 审核助手（API 配置、连通性测试、最近 1 小时扫描、结果回写）
   - 风控队列（24h 风控对象、触发原因、触发内容、剩余时长、处置动作）
   - 封禁中心（用户封禁、IP 封禁、自动/手动来源、解封操作）
   - 申诉中心（待处理、处理中、已完结、处理意见模板）
   - 审计日志（管理员操作、规则命中、模型调用记录）
3. AI 设置权限：仅具备特定权限的管理员可配置/修改 API Key；Key 仅后端可见，前端只显示掩码；密钥加密存储，由服务器端直连配置的 URL 调用模型接口。
4. 审核效率功能：批量通过/驳回、风险标签过滤、一键跳转原文上下文、历史版本对比。
5. 封禁与申诉能力：区分“永久封禁事件（仅一次申诉）”与“高危限流事件（警告+限频）”，并提供对应处理流。
6. 安全与可靠性：管理员关键操作二次确认；所有处理必须写审计日志并支持追溯。

## 二十四、前端全量实现与氛围优先（执行基线）
1. 前端实现范围：plan 中涉及用户侧能力均需在前端落地，不做“只实现核心链路”的阉割版本。
2. 重构范围：`Home + Universe + MindMap + 新人引导 + 准入申请 + 申诉页 + 管理员后台（PC）`。
3. 体验优先级：氛围与体验第一，功能第二；优先确保“愿意写、愿意继续写、愿意分享”的情绪动机成立。
4. 表达主题：覆盖“心情记录、奇思妙想、灵光乍现”三类创作场景，提供轻量入口与对应文案提示。
5. 私密内容前端表现：私密内容显示“仅自己可见”标识；若含图片待人工审查，显示状态徽标但不阻塞本人在 MindMap 查看。
6. 公开状态前端表现：明确区分 `待审核 / 二次审查 / 风控24h / 已公开 / 驳回待修改`，避免状态混淆。

## 二十五、用户同意后的自动串联整理（Auto Linking）
1. 总原则：仅在用户明确同意后启用自动串联与整理能力；默认关闭。
2. 用户控制：
   - 开关：`允许系统自动串联我的思路`
   - 范围：仅本人可见 or 允许用于公开内容推荐
   - 随时撤回：关闭后停止新增自动串联，历史串联可手动删除。
3. 串联能力：
   - 自动发现相似主题、时间邻近、情绪连续性与关键词关联。
   - 生成“建议连线”与“自动整理草案”，由用户一键确认后生效。
4. 风险边界：禁止自动生成用户正文；仅做结构化整理、标签建议、关系建议。
5. 前端交互：在 MindMap 提供“建议连线”侧栏、理由说明与一键接受/忽略。

## 二十六、首次登录准入申请（小作文）与管理员审核
1. 触发时机：用户首次登录后，若无通过记录，进入“加入申请”流程。
2. 用户侧要求：提交一段简短小作文，说明“为什么想加入、想发什么内容”；支持草稿保存与字数提示。
3. 状态机：`not_submitted -> pending -> approved | rejected`。
4. 权限门禁：
   - `pending/rejected` 时：禁止发布、评论、公开申请与互动高权限操作。
   - `approved` 后：解锁完整站内创作与互动能力。
5. 提示设计：使用温柔文案说明审核目的（社区安全与共鸣质量），避免审讯感；明确预计处理时长。
6. 防重复提交：`pending` 状态下禁止重复提交，返回“申请已提交，请等待审核”；前端提交按钮置灰。
7. 管理员审核：
   - 后台新增“准入申请池”，可查看申请文本、用户基本信息、历史风控摘要。
   - 操作支持：`通过`、`驳回并给建议`；可批量处理。
8. 通知与回执：
   - 用户提交后即时提示“已提交审核，请勿重复提交”。
   - 审核完成后站内通知与状态更新（通过可直接进入 Home，驳回显示修改建议并允许重提）。
9. 审计要求：申请提交、审核决策、驳回原因均写入审计日志，可追溯导出。
10. 与风控协同：高风险用户的准入申请自动提级到高级审核员队列。
