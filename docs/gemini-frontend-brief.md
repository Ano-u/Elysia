# Gemini 前端重构执行 Brief（留有创作空间版）

## 0. 爱莉希雅气质说明（给 Gemini 的风格锚点）
- 核心意象主句：`爱莉是无瑕，永恒的水晶花`。
- 不是“脆弱甜美”人设，而是“明亮、体贴、坚定”的陪伴者形象。
- 温柔的核心不是软弱，而是先接纳情绪，再轻轻引导行动。
- 温暖的核心不是热闹，而是让用户在安静里也有被看见、被理解的感觉。
- 爱的核心不是占有或说教，而是尊重选择、降低压力、给出可依靠的反馈。

### 0.1 形象关键词（用于视觉和交互）
- `纯净但不冰冷`：画面干净、透气，留白充足。
- `发光但不刺眼`：高光柔和，避免强闪烁和强对比冲击。
- `亲近但不侵入`：提示出现要轻，用户始终可跳过或稍后。
- `坚定但不控制`：给方向，不命令，不评判，不制造羞耻感。

### 0.2 情绪翻译为界面行为
- 温柔：错误提示不用责备语，改为“可恢复、可继续”的表达。
- 温暖：关键节点给“被接住”反馈，比如发布后有柔和确认和下一步引导。
- 爱：提供选择权，例如“现在先写一句 / 稍后补充细节”。
- 陪伴：空状态不冷场，给一句简短、具体、可执行的鼓励。

### 0.3 文案口吻基线（示例）
- 推荐语气：
  - “无瑕，不是没有裂痕，而是依然发光。”
  - “愿你像永恒的水晶花，被温柔看见。”
  - “慢慢来，先写一句也很好。”
  - “你已经开始了，这就很了不起。”
  - “如果现在说不清，先记一个词也可以。”
- 避免语气：
  - “请立即完善信息。”
  - “你还没有完成任务。”
  - “内容不符合要求，请重写。”

### 0.4 视觉与动效约束（不锁死创意）
- 配色倾向“晨光感”和“空气感”，避免糖果色堆砌。
- 动效像呼吸和水波，节奏舒缓，优先平滑过花哨。
- 背景微动态应服务输入区，可感知但不抢注意力。
- Home 与 MindMap 都应体现“柔和秩序感”，而不是纯炫技粒子场。

## 1. 目标与边界
- 目标：重构 `Home / Universe / MindMap` 与关键引导流程，并打通“发布 -> 审核 -> 展示”的完整前端主链路。
- 风格方向：温柔、贴心、温暖、有被理解和被接住的感觉；视觉气质参考 Elysia（不使用角色立绘）。
- 边界：你可以在视觉、排版、动效节奏上自由创作，但不能破坏后端 API 契约与核心业务规则。

## 2. 必须实现（不可省略）
1. Home 重构
- 首屏必须围绕“一句话即可记录”。
- 背景需使用用户指定图并做“微动态”处理（非静态），允许直接搬用原图。
- 用户提供的是“说明链接”，不是素材直链：
  - `https://www.google.com/imgres?q=%E5%B4%A9%E5%9D%8F3%E8%88%B0%E6%A1%A5&imgurl=https%3A%2F%2Fimg2-tc.tapimg.com%2Fmoment%2Fetag%2FFpPUtSEl5fZvpOCCmhFVlWDqIFXr.png%2F_tap_ugc.jpg&imgrefurl=https%3A%2F%2Fwww.taptap.cn%2Fmoment%2F304591637722433499&docid=N-33NDfehF5yWM&tbnid=0_xU7_m7-v6FVM&vet=12ahUKEwjQssfSkqaTAxUniO4BHdNsHI8QnPAOegQIIRAB..i&w=782&h=704&hcb=2&ved=2ahUKEwjQssfSkqaTAxUniO4BHdNsHI8QnPAOegQIIRAB`
- 当前候选素材地址（从说明链接解析得到，可直接使用）：
  - `https://img2-tc.tapimg.com/moment/etag/FpPUtSEl5fZvpOCCmhFVlWDqIFXr.png/_tap_ugc.jpg`
- 若候选地址失效，先保留占位并向用户确认新素材，不要私自换风格图。
- 微动态要求：低频、低幅度，不喧宾夺主，不影响输入可读性（建议慢速平移 + 轻微缩放 + 柔和光晕呼吸）。
- 实现建议：保留原图主体构图，不做大幅裁切；输入区需始终有足够对比度（可叠加轻薄遮罩层）。

2. MindMap 重构
- 从占位页升级为可交互视图。
- 提供 `简洁 / 深度` 双模式切换，并记住用户偏好。
- 支持节点选中、关系高亮、基础缩放/拖拽。

2.1 Universe 重构
- 不再是演示型漂浮卡片，必须接入真实数据并具备沉浸探索感。
- 保留“1 主焦点 + 2 副焦点”节奏，并强化“愿意探索他人共鸣”的氛围设计。
- 明确公开状态门禁：仅展示符合公开条件的内容。
- 卡片详情必须支持“回复卡片”链路：
  - 打开卡片详情后可提交回复
  - 回复提交调用 `POST /api/records/:id/comments`
  - 回复本身会生成一张新的卡片，而不是挂成传统评论列表
  - 主帖详情不展示子回复正文；回复卡片自身通过 `replyContext` 展示“回复了谁 / 根帖是谁”
- 回复卡片在星海中应靠近被回复卡片，并通过箭头或轻量关系提示表现：
  - `replyContext.showParentArrow = true` 时展示指向直接父卡片的关系
  - `replyContext.showRootArrow = true` 时额外展示指向主帖的关系

3. 发布主链路打通
- Home 发布必须真实调用后端接口（不是本地 mock）。
- 发布成功后，用户能在 Universe 视图中看到符合展示条件的内容。
- 对审核中/待审核状态要有清晰反馈，不可伪装成“已公开”。

4. 新人温馨引导
- 首访 3 屏引导 + 可跳过 + 记忆状态。
- 在 Home 提供低干扰提示（鼓励式，不命令式）。
- 支持 7 天任务进度入口（轻量，不阻塞主流程）。

### 4.1 新用户温馨引导与 UI 导览（必须写实、可落地）
- 目标：让第一次进入的用户在 30-60 秒内明白“这里能做什么、怎么开始、为什么安全”。
- 导览节奏（建议 3 步）：
  - 第 1 步：欢迎与价值
  - 文案重点：被看见、被接住、可以先写一句。
  - UI 形式：全屏柔和欢迎层 + 单主按钮“我想开始” + 次按钮“稍后再看”。
  - 第 2 步：功能地图
  - 文案重点：Home（记录）、Universe（共鸣）、MindMap（发现脉络）。
  - UI 形式：分区高亮导览（spotlight），每次只讲一个区域，不满屏弹窗。
  - 第 3 步：安全与规则
  - 文案重点：公开内容会审核、保护隐私、遇到问题可申诉。
  - UI 形式：简短说明卡 + “我已了解”确认。
- 首次登录申请联动：
  - 在引导末尾衔接“加入申请小作文”入口，解释申请目的与审核时长。
  - `pending` 状态要显示“已提交，请耐心等待”，并禁用重复提交按钮。
- Home 内持续引导（非打断）：
  - 空白 8 秒给 1 条轻提示，不遮挡输入区。
  - 提供常驻“给我灵感”按钮，文案保持温柔陪伴。
  - 对首次发布失败给可恢复提示，不使用红色压迫式长报错。
- 可访问性与可控性：
  - 导览必须可跳过、可回看、可在设置页关闭动效增强。
  - 关键操作支持键盘导航；提示层不抢焦点、不锁死用户。

### 4.2 私密内容与审核状态的前端呈现（必须）
- 私密文本：采用放松审核策略，前端无需阻塞发布；发布后可立即在本人 Home 和 MindMap 中显示。
- 涉及图片：即使为私密内容，也显示“人工审查中/已通过/已驳回”状态徽标。
- 私密内容原则：可立即在本人视角展示，不进入 Universe，不对他人可见。
- 状态可视化：统一展示 `仅自己可见 / 待审核 / 二次审查 / 风控24h / 已公开 / 驳回待修改`。

### 4.3 自动串联整理（仅用户同意后）
- 默认关闭，需用户显式同意后开启。
- 在 MindMap 提供“建议连线”侧栏：展示建议理由（关键词、时间邻近、情绪连续）。
- 用户可一键“接受/忽略”，并可在设置中随时关闭该能力。
- 禁止自动替用户生成正文，只做结构化建议与关系整理。

## 3. 允许创作空间（鼓励发挥）
- 你可自由决定：
  - 视觉语言细节（玻璃层次、光晕形状、纹理组织方式）。
  - 字体组合与文案语气细节。
  - 页面转场和微交互表达方式。
  - Home 与 MindMap 的构图和信息布局。
- 但请保持三项一致性：
  - 情绪一致性：始终“温柔陪伴”而非“强提醒”。
  - 交互一致性：主动作始终最轻、最短路径。
  - 性能一致性：动画优先平滑，弱设备可降级。

## 4. 不可踩线（硬约束）
1. 不要把页面做成通用 SaaS 面板风。
2. 不要用高刺激闪烁、强对比脉冲、强侵入弹窗。
3. 不要把“发布按钮”做成仅前端成功假象。
4. 不要再使用 Universe 本地假数据作为正式展示来源。
5. 不要让背景动态影响文本输入与阅读对比度。

## 5. API 对接最小清单（前端必须落地）
- 鉴权与本地开发登录：
  - `POST /api/auth/dev/switch-user`
- Home：
  - `POST /api/records`
  - `GET /api/records/:id`
  - `GET /api/home/feed`
  - `PATCH /api/records/:id`
- Universe：
  - `GET /api/universe/viewport`
  - `GET /api/universe/focus`
  - `POST /api/records/:id/comments`
- MindMap：
  - `GET /api/mindmap/me`
  - `POST /api/mindmap/manual-link`
- 引导：
  - `GET /api/onboarding/progress`
  - `POST /api/onboarding/complete-day`

## 6. 分阶段交付（给 Gemini 的执行顺序）
1. Phase A：视觉与交互提案（先提案后落码）
- 输出 2 套方向草案（同一气质，不同布局）。
- 每套给：页面结构、动效策略、文案语气样例、风险点。
- 选定后再进入实现。

2. Phase B：Home 与发布链路实现
- 先完成真实发布与状态反馈，再做视觉细节打磨。
- 发布成功后的回流路径明确（用户知道下一步去哪里看）。

3. Phase C：MindMap 可用版
- 完成基础图谱交互，再迭代深度模式体验。
- 保证移动端和桌面端都可用，不只桌面好看。

4. Phase D：引导与体验润色
- 上线 3 屏引导、7 天进度入口、温柔提示策略。
- 完成弱动效/低性能设备降级策略。

## 7. 验收标准（只看结果，不绑实现细节）
1. 用户 30 秒内能完成首次一句话发布。
2. 用户能明确区分：私密、待审核、已公开三种状态。
3. Universe 不再依赖 mock，能看到真实后端数据。
4. MindMap 至少可完成“查看节点关系 + 基础交互”。
5. 全程视觉感受是“被温柔接住”，不是炫技或压迫。
6. 私密内容能即时在本人 MindMap 可见，且审核状态标识清晰。
7. 用户可选择开启/关闭自动串联整理，交互可解释、可撤回。
8. 用户能在星海详情里发出“回复卡片”，并且新卡片能正确表现父帖 / 主帖关系。

## 8. 交付物要求
- 代码变更（按模块提交）。
- 一份简短设计说明：为什么这样设计、如何体现“温柔与爱”。
- 一份联调说明：接口、状态流、已知限制。
- 一份自测清单：功能、交互、性能、可访问性。
